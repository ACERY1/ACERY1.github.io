简单理解和使用JavaScript闭包

ACERY 12月 18, 2017
  • JavaScript Tricks

从真正接触学习JavaScript这门语言的时候，就已经在红宝书(<<JavaScript高级程序设计>>)里看到了闭包(closure)，每次看完都以为自己理解了，但是真正要用的时候却想不到，或者乱用。从根本来说，我认为把闭包看做一个技术技巧比看做语言的特点更为实用和容易掌握。

首先，放一个官方释义

Closures (闭包)是使用被作用域封闭的变量，函数，闭包等执行的一个函数的作用域。

OK，inner函数就是闭包

那么再看看之前那句定义（我帮忙断了句）：

Closures (闭包)是/使用/被作用域封闭的/变量，函数，闭包等/执行的一个函数的作用域。

  • “被作用域封闭的变量，函数，闭包等” 指的就是我们的wrap函数里面的变量，函数等
  • ”执行的一个函数的作用域“指的就是inner函数

我们的inner函数因为函数作用域链的关系能够访问_lastTime变量，而inner函数因为被当做wrap函数的返回值了，所以能够在全局访问到inner函数。所以效果就是

通过inner函数作为媒介在全局间接访问到了wrap函数内部的变量。

那么这个媒介，就是我们常说的闭包

为什么要使用闭包

  • 想要封装一个功能，但是不想污染全局变量。
  • 想要完成对象的私有属性。
  • 想要让某些变量不因函数执行完成而被销毁。

闭包是实现模块化的一个关键，他让变量的生命周期超出其应有的作用域生效时间(因为其被写到了外部),同时，也让变量在私有化的同时能够特殊的方法去访问。在一定意义上是有点类似于private的。

从存储历史来理解闭包

仍然为了解决这个问题：我们想让此次函数的执行过程中能够使用上一次函数执行后的某个变量的数据。即保存了上一次函数执行后的状态。

但我们都知道，函数的一次执行后，所有函数内部的变量等都会被销毁，根本无法保存状态，所以闭包为我们做到了。保存一个State不因函数的执行而被销毁。

我们可以抽象的将函数看成如上图，每一次函数的执行会去读取和更新State，然后除State之外所用内存会在函数执行完成后被销毁，而state本身却仍然属于Function作用域的。

所以，使用闭包的原因是：

让某函数的多次执行都能访问到同一变量且该变量不污染全局。

当需要实现类似的功能的时候，请先想想是否可以问闭包。

闭包无处不在(使用场景)

  • 函数节流/节流要记录上一次函数触发的时间。
  • 模拟私有方法
  • 某些事件触发更改的数据想要封装得更好
  • 某些框架底层变量只想通过固定的方法去访问

使用闭包很大程度上能够降低变量的可访问性，这也是我们使用闭包的根本原因。

闭包的坏处

如果我们不小心在循环里面创建了闭包，那么我们这个程序所占用的系统内存将会无限增加，原因就是因为闭包访问到的上级作用域变量是不被销毁的，导致GC无法对其进行自动回收，从而内存越占越多。当然消除的办法也很简单，手动将闭包访问的变量设为null。

小结

我们使用闭包的主要原因是为了变量的私有化和延长其生命周期。实现方式就是利用函数作用域链通过闭包的媒介访问到某上级作用域里的数据。
